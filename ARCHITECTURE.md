# 数学建模智能体 — 技术架构文档

> **版本**: v0.1 (2026-07-06)

## 1. 项目定位

构建一个辅助解决数学建模问题的多智能体系统。核心目标有两个层次：

- **实战辅助**: 面对具体建模问题时，能够输出结构化的建模方案（问题分析、模型选择、求解步骤、代码实现、论文框架）
- **能力培养**: 通过教学模式引导用户理解建模思维，为参加数学建模竞赛（国赛、美赛等）做准备

附带功能（低优先级）：结构化学习笔记生成系统，支持多格式导出、交互答疑、自定义模板和溯源引用。

## 2. 系统总体架构

### 2.1 多智能体分层架构

```
┌─────────────────────────────────────────────┐
│                 用户交互层                     │
│          (输入问题 / 选择模式 / 查看输出)        │
└─────────────────┬───────────────────────────┘
                  │
┌─────────────────▼───────────────────────────┐
│              主智能体 (Orchestrator)           │
│  ┌─────────────┐  ┌───────────────────────┐  │
│  │  模式路由器   │  │  问题分类与规划引擎     │  │
│  └─────────────┘  └───────────────────────┘  │
└──┬──────┬──────┬──────┬──────┬───────────────┘
   │      │      │      │      │
┌──▼──┐┌──▼──┐┌──▼──┐┌──▼──┐┌──▼──┐
│分析  ││建模  ││求解  ││验证  ││写作  │  ← 子智能体
│Agent ││Agent ││Agent ││Agent ││Agent │
└──┬──┘└──┬──┘└──┬──┘└──┬──┘└──┬──┘
   │      │      │      │      │
┌──▼──────▼──────▼──────▼──────▼───────────────┐
│              知识库层 (Knowledge Base)          │
│  ┌──────────┐ ┌───────────┐ ┌─────────────┐  │
│  │方法卡片库 │ │真题论文库  │ │模板套路库    │  │
│  │(地基)     │ │(主体)      │ │(脚手架)      │  │
│  └──────────┘ └───────────┘ └─────────────┘  │
└──────────────────────────────────────────────┘
```

### 2.2 设计原则

- **主 agent 负责编排，子 agent 负责执行**: 主 agent 不直接产出建模内容，而是理解问题、规划步骤、调度子 agent、整合输出
- **动态编排而非固定流水线**: 主 agent 根据问题特征和当前进展动态决定下一步调用哪个子 agent，支持迭代回退（如验证不通过时回到建模阶段）
- **知识库三层联动**: 任何子 agent 在工作时都可以同时检索三层知识库，确保输出既有理论依据又有实战参考

## 3. 主智能体设计

### 3.1 双模式系统

主智能体支持两种工作模式，可随时切换：

#### 教学模式 (Plan Mode)

| 维度 | 说明 |
|------|------|
| **核心目标** | 引导用户理解建模思维，培养独立解题能力 |
| **交互方式** | 苏格拉底式提问，逐步引导，不直接给答案 |
| **输出特征** | 思维过程展开、方案对比分析、原理讲解、决策推理 |
| **典型输出** | "这个问题的决策变量是什么？""如果目标函数是非凸的，你会想到哪些方法？" |
| **适用场景** | 日常训练、备赛学习、理解新模型 |

#### 方案输出模式 (Execute Mode)

| 维度 | 说明 |
|------|------|
| **核心目标** | 输出可直接执行的完整建模方案 |
| **交互方式** | 接收问题，结构化输出方案，用户确认后执行 |
| **输出特征** | 问题分析、模型假设、数学建模、求解步骤、代码框架、论文大纲 |
| **典型输出** | 完整的 Markdown 方案文档，含公式、代码、图表 |
| **适用场景** | 实战解题、竞赛冲刺、方案评审 |

#### 模式切换机制

```
用户输入
   │
   ▼
[模式路由器] ── 检查当前模式 ──┬── 教学模式 → 调用教学策略子流程
                              │
                              └── 方案模式 → 调用方案生成子流程
```

用户可以随时通过指令切换模式（如 `/teach`、`/solve`），主 agent 也可以根据上下文建议切换（如用户在方案模式下反复追问"为什么"，主动建议切换到教学模式深入讲解）。

### 3.2 问题分类与规划引擎

主 agent 接收到用户输入后，首先进行问题分类：

**分类维度**:

- **问题类型**: 优化问题 / 预测问题 / 评价问题 / 分类问题 / 拟合问题 / 图论问题 / 博弈问题 / 排队论问题 / 微分方程问题 / 综合问题
- **复杂度评估**: 单模型可解 / 需要组合模型 / 需要创新性建模
- **数据依赖**: 纯理论推导 / 有给定数据 / 需要自行采集或生成数据

**规划输出**: 根据分类结果生成执行计划，决定调用哪些子 agent、以什么顺序、从知识库的哪些层检索信息。

## 4. 子智能体设计

### 4.1 子 agent 划分（按建模流程阶段）

| 子智能体 | 职责 | 输入 | 输出 |
|---------|------|------|------|
| **问题分析 Agent** | 理解题意，提取关键信息，明确问题边界 | 原始题目文本 | 结构化问题分析（已知条件、求解目标、约束条件、隐含假设） |
| **模型构建 Agent** | 选择或设计数学模型 | 问题分析结果 + 知识库推荐 | 模型假设、数学公式、模型结构 |
| **求解计算 Agent** | 编写求解代码，执行计算 | 模型描述 | 算法设计、Python/MATLAB 代码、计算结果 |
| **验证分析 Agent** | 模型验证与鲁棒性分析 | 模型 + 求解结果 | 灵敏度分析、误差分析、模型改进建议 |
| **论文写作 Agent** | 生成规范论文 | 全流程输出 | 结构化论文（摘要、问题重述、模型假设、建立与求解、结果分析、参考文献） |

### 4.2 子 agent 协作流程

子 agent 之间不是简单的线性流水线，而是由主 agent 动态编排：

```
[主 Agent 规划]
       │
       ├── 1. 问题分析 Agent ──────────────► 问题分析报告
       │                                         │
       ├── 2. 模型构建 Agent (检索知识库) ──► 模型方案
       │       │                                  │
       │       ├── 2a. 如果模型需要组合 ──► 多次调用，整合
       │       │
       ├── 3. 求解计算 Agent ──────────────► 代码 + 结果
       │                                         │
       ├── 4. 验证分析 Agent ──────────────► 验证报告
       │       │                                  │
       │       ├── 如果不通过 ──► 回退到步骤 2 或 3
       │       │
       └── 5. 论文写作 Agent ──────────────► 论文草稿
```

**关键规则**:

- 每个子 agent 的输出会作为上下文传递给后续子 agent
- 主 agent 有权在任何阶段要求子 agent 重新输出（如验证不通过时回退）
- 教学模式下，每个子 agent 的输出前会先经过"引导式转化"，变成提问而非结论

## 5. 知识库设计

### 5.1 三层架构

```
┌─────────────────────────────────────────────┐
│           第三层: 模板套路库 (脚手架)           │
│    思维框架级模板，指导分析路径而非填空          │
├─────────────────────────────────────────────┤
│           第二层: 真题论文库 (主体)             │
│    结构化拆解的竞赛论文，支持特征检索            │
├─────────────────────────────────────────────┤
│           第一层: 方法卡片库 (地基)             │
│    高频方法切片，含原理、适用条件、代码           │
└─────────────────────────────────────────────┘
```

### 5.2 第一层：方法卡片库

**设计原则**: 不塞整本教材，只抽取竞赛最高频的方法切片。筛选标准是该方法在建模中被用到的频率和被用错的频率。

**单张卡片结构**:

```yaml
method_card:
  id: "mc_001"
  name: "蒙特卡洛模拟"
  category: ["数值方法", "随机模拟"]
  
  # 核心原理
  principle: |
    通过大量随机抽样来近似求解确定性或随机性问题的数值方法。
    核心依据是大数定律：当样本量足够大时，样本均值趋近于期望值。
  
  # 核心公式
  formulas:
    - name: "蒙特卡洛积分估计"
      latex: "\\hat{I} = \\frac{V}{N} \\sum_{i=1}^{N} f(x_i)"
      description: "V为采样域体积，N为样本数，x_i为均匀随机采样点"
  
  # 适用条件
  applicable_when:
    - "问题可以转化为期望值或积分计算"
    - "解析解难以获得或不存在"
    - "高维问题（传统数值方法维度灾难）"
  
  # 不适用条件
  not_applicable_when:
    - "需要精确解析解"
    - "计算预算极其有限且维度较低（此时用确定性数值方法更优）"
  
  # 典型建模场景
  typical_scenarios:
    - "复杂系统可靠性评估"
    - "金融衍生品定价"
    - "排队系统性能仿真"
    - "不规则区域面积/体积计算"
  
  # 常见误用
  common_mistakes:
    - mistake: "样本量过小导致结果不稳定"
      solution: "进行收敛性检验，逐步增大样本量直到结果稳定"
    - mistake: "忽略随机数质量"
      solution: "使用高质量伪随机数生成器，必要时用拟随机序列（如 Sobol 序列）"
  
  # 代码片段
  code_snippets:
    - language: "python"
      description: "用蒙特卡洛方法估算圆周率"
      code: |
        import numpy as np
        def estimate_pi(n_samples=1000000):
            x = np.random.uniform(-1, 1, n_samples)
            y = np.random.uniform(-1, 1, n_samples)
            inside = (x**2 + y**2) <= 1
            return 4 * np.sum(inside) / n_samples
  
  # 关联卡片
  related_cards: ["mc_latin_hypercube", "mc_quasi_random", "mc_importance_sampling"]
  
  # 关联真题
  related_papers: ["2019A_热力管网", "2021C_无人机调度"]
```

**初期收录范围** (按优先级排序):

| 优先级 | 类别 | 方法示例 |
|-------|------|---------|
| P0 | 优化方法 | 线性规划、整数规划、动态规划、遗传算法、模拟退火、粒子群 |
| P0 | 预测方法 | 时间序列(ARIMA)、灰色预测、回归分析、神经网络 |
| P0 | 评价方法 | AHP、TOPSIS、熵权法、模糊综合评价、DEA |
| P1 | 统计方法 | 假设检验、极大似然估计、贝叶斯推断、蒙特卡洛模拟 |
| P1 | 图论方法 | 最短路径、网络流、最小生成树、图着色 |
| P2 | 微分方程 | 常微分方程建模、偏微分方程、有限差分法 |
| P2 | 机器学习 | 聚类(K-Means/DBSCAN)、分类(SVM/RF)、PCA与特征值 |
| P2 | 其他 | 排队论、博弈论、插值与拟合 |

### 5.3 第二层：真题论文库

**设计原则**: 不是存 PDF 原文，而是对每篇论文做结构化拆解，使其可被 agent 检索和参考。

**单篇论文结构**:

```yaml
paper:
  id: "paper_2023A_01"
  year: 2023
  competition: "国赛"        # 国赛 / 美赛 / 研赛 / ...
  problem_id: "A"
  title: "xxx"
  
  # 问题分类标签
  tags:
    problem_type: ["优化", "调度"]
    core_models: ["混合整数规划", "遗传算法"]
    techniques: ["多目标优化", "启发式算法"]
  
  # 核心内容拆解
  analysis:
    problem_summary: "一句话概括问题本质"
    key_assumptions: ["假设1", "假设2", "..."]
    decision_variables: "决策变量是什么"
    objective: "目标函数是什么"
    constraints: "主要约束条件"
    
  # 模型与求解
  model:
    approach: "建模思路概述"
    innovation: "该论文的建模创新点"
    solution_method: "求解算法描述"
    
  # 评价
  evaluation:
    strengths: ["优点1", "优点2"]
    weaknesses: ["不足1", "不足2"]
    lessons: "可以学到什么"
  
  # 元数据
  source: "原始论文文件路径或链接"
  quality_rating: 4  # 1-5, 论文质量评级
```

**检索策略**: 当面对新问题时，agent 通过问题类型标签 + 核心模型标签进行相似题检索，而非全文匹配。

### 5.4 第三层：模板套路库

**设计原则**: 粒度是**思维框架级别**，而非"填空题"。模板指导分析路径，不替代思考。

**模板示例 — 优化问题分析框架**:

```yaml
template:
  id: "tpl_optimization"
  name: "优化问题分析框架"
  applicable_to: ["优化问题", "调度问题", "分配问题"]
  
  steps:
    - step: 1
      name: "问题识别"
      guiding_questions:
        - "需要优化的目标是什么？（最大化/最小化）"
        - "决策变量有哪些？"
        - "存在哪些约束条件？（等式/不等式）"
        - "目标函数和约束是线性还是非线性？"
    
    - step: 2
      name: "模型分类决策"
      decision_tree:
        - "所有函数线性 + 变量连续 → 线性规划 (LP)"
        - "所有函数线性 + 变量含整数 → 混合整数规划 (MIP)"
        - "目标/约束非线性 + 凸性 → 凸优化"
        - "目标/约束非线性 + 非凸 → 启发式算法 (GA/SA/PSO)"
        - "多阶段决策 → 动态规划 (DP)"
        - "多个目标 → 多目标优化 (NSGA-II / 加权求和)"
    
    - step: 3
      name: "模型建立"
      checklist:
        - "明确写出目标函数的数学表达式"
        - "列出所有约束条件"
        - "检查决策变量的取值范围和物理意义"
        - "简化假设是否合理（需后续验证）"
    
    - step: 4
      name: "求解与验证"
      checklist:
        - "选择求解器或编写算法"
        - "小规模算例验证模型正确性"
        - "灵敏度分析：关键参数变化对结果的影响"
        - "模型假设放松后的鲁棒性"
```

### 5.5 三层联动机制

当主 agent 面对一个新问题时，三层知识库的联动流程：

```
用户输入新问题
       │
       ▼
[问题分类器] ──► 识别问题类型: "多目标优化问题"
       │
       ├──► 方法卡片库: 推荐相关方法
       │    (NSGA-II、加权求和法、ε-约束法、帕累托前沿...)
       │
       ├──► 真题论文库: 检索相似赛题
       │    (2019A、2021C、2023B... 标签匹配)
       │
       └──► 模板套路库: 调出分析框架
            (优化问题分析框架)
       │
       ▼
[主 Agent 整合] ──► 带着方法知识 + 真题参考 + 分析框架，开始编排子 agent
```

## 7. 开发路线图（初步）

> 待需求对齐后细化。

**Phase 0 — 知识库基建**
- 确定方法卡片、论文库、模板库的存储格式和目录结构
- 录入 P0 级别的方法卡片（约 20-30 张）
- 结构化拆解 5-10 篇经典真题论文
- 编写 3-5 个核心问题类型的分析模板

**Phase 1 — 核心 Agent**
- 实现主 agent 框架（模式切换 + 问题分类）
- 实现子 agent 基础版（先做分析 agent 和建模 agent）
- 打通知识库检索链路

**Phase 2 — 完整流程**
- 补全求解、验证、写作子 agent
- 实现动态编排（含回退机制）
- 三层知识库联动调优

**Phase 3 — 体验打磨**
- 教学模式深度优化（引导策略、难度分级）
- 论文输出格式化（符合竞赛提交要求）
- 附带功能：学习笔记系统（复用知识库和输出框架）
