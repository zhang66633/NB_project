## 一、记忆管理

### 1.1 核心范式: 整体重写 vs 原子事实提取

Agent Xi v1 用"原子事实提取"——每轮对话从用户输入中抽取离散事实存入 SQLite。实际运行暴露了严重问题: 一次对话产生 14 条近乎重复的记忆（"用户喜欢像素风" / "用户偏好像素风格" / "用户表示喜欢像素风游戏"），没有去重、没有合并、没有衰减。

v2 改为"整体重写"范式: 用户画像是一个 Markdown 文件 (`service.md`)，每次会话结束时 LLM 读旧文档 + 本次对话，输出全新文档。结构性地解决了重复和矛盾问题。

**设计要点:**

```
会话结束
  → 格式化最近 20 轮对话为纯文本
  → 读取当前 service.md
  → 发给 LLM: "这是旧档案，这是本次对话，重写档案"
  → 校验输出（非空、>=10字符、有标题行）
  → 保存旧版本快照 → 写入新文件
```

**重写 prompt 的关键约束（直接可用的模式）:**

- 过滤启发式: "下次醒来不知道这件事，会做错事吗？答 yes 才写进去"
- 定位: "状态档案，不是日志"——记录用户是什么样的人，不是今天发生了什么
- 字数预算: 300 字以内（防止 LLM 写成小作文）
- 正反例: 给 3 个好例子 + 3 个坏例子，比纯指令有效 10 倍
- 逃逸条件: "如果没有新认知，原样返回"（防止 LLM 为了"有输出"而瞎改）

**适用场景判断:**
- 用户画像/偏好/长期状态 → 整体重写（文档范式）
- 具体事件/对话摘要/可检索的历史 → 向量存储（情景记忆）
- 两者不互斥，Agent Xi 同时用了两层

### 1.2 情景记忆: 向量检索

用 LanceDB（本地嵌入式向量数据库）存储显式记忆条目。只在用户明确说"记住这个"或 `/remember` 命令时写入，不做隐式提取。

**关键决策:**
- 嵌入用 summary（如果有）而非原文 → 索引语义更精准
- 检索 top_k=3，注入 system prompt 尾部（不是作为单独 message）
- 无时间衰减（TODO），靠整体重写层自然淘汰过时信息
- 分数转换: LanceDB 返回 distance，用 `1.0 - distance` 转为相似度

**math_agent 适用点:** 多智能体系统中，每个子 Agent 的"方法卡片库"和"真题论文库"本质上就是向量检索。区别在于知识库是静态预建的，不是对话中动态积累的。建议: 静态知识库用预计算 embedding + 元数据过滤（按题型/方法分类），动态经验（"上次用灰色预测效果不好"）用情景记忆。

### 1.3 人设三段式

```
config/
  identity.md    — 我是谁（名字、本质、气质、Traits 自述段）
  personality.md — 我怎么做事（Core Truths + 反面例子 + 授权 + 硬边界）
  service.md     — 我为谁服务（用户画像，LLM 自动重写）
```

**Prompt 组装顺序:** identity → personality → 能力列表（动态）→ 记忆指令 → service

**设计模式提取（来自智谱学习搭子）:**
- Core Truths 格式: 粗体原则 + "不是...而是..." 反面例子 + 明确授权（"你可以..."）
- Traits 自述段: 第一人称写"我叫什么，我的风格是..."，比第三人称描述更有代入感
- 记忆连续性叙事: "每次会话你都会醒来，这些文件就是你的记忆"
- 自修改透明: "如果你修改了这些文件，必须告知用户"

**math_agent 适用点:** 多智能体中每个子 Agent 也需要角色定义。建议 Orchestrator 有完整三段式，子 Agent 只需 identity + 能力约束（不需要 service，因为用户画像由 Orchestrator 统一持有）。

### 1.4 快照与版本控制

每次重写 service.md 前，旧版本存为 `{filename}.{YYYYMMDD_HHMMSS}.bak`，保留最近 20 份。

**教训:** 没有快照 = 一次 LLM 幻觉就永久丢失用户画像。有快照 = 可以回滚、可以 diff、可以审计"LLM 到底改了什么"。

**math_agent 适用点:** 多智能体协作中，Orchestrator 的"问题理解文档"会被反复修改（分析 Agent 补充、验证 Agent 修正）。每次修改前保存快照，出错时可回退到上一个正确版本。

## 二、上下文管理

### 2.1 Token 预算分配

```
总预算 = max_context_tokens (128K)
- reserved_output_tokens (4K)     ← 留给模型输出
- system_prompt tokens            ← 人设 + 能力 + 记忆指令
- memory_context tokens           ← 召回的情景记忆
- tools_schema tokens             ← 工具定义 JSON
= 可用于历史消息的预算
```

**关键实现:**
- 固定开销（system + tools）先扣除，剩余给历史
- 保底 1000 tokens 给历史（防止 system prompt 太长导致完全没历史）
- 双重限制: token 预算 + 轮次硬上限（50轮 = 100条消息），取更严的

### 2.2 截断策略: 贪心逆序填充

```python
# 从最新消息往回贪心装，装满为止
for msg in reversed(candidates):
    if used + msg_tokens > budget and already_have_messages:
        break
    include(msg)
```

**特点:**
- 最新消息永远保留（保证当前轮用户输入不丢）
- 最旧消息先被丢弃
- 无摘要——直接丢弃，不做"把旧消息压缩成摘要"
- 完整历史仍在内存/磁盘中，只是不发给 LLM

**为什么不做摘要:**
- 摘要本身需要额外 LLM 调用（延迟 + 成本）
- 摘要质量不可控（可能丢关键细节）
- 对于 128K 窗口，50 轮对话通常装得下，截断很少触发
- 简单正确 > 复杂聪明

**math_agent 适用点:** 多智能体系统中上下文更复杂——Orchestrator 需要持有所有子 Agent 的输出摘要。建议:
- 子 Agent 返回给 Orchestrator 的结果做结构化摘要（不是全文）
- Orchestrator 的 context 中只放"当前阶段相关的子 Agent 输出"，不是全部历史
- 长程建模任务（跨多轮）用"工作文档"模式: 维护一个持续更新的问题状态文档，每轮只把文档 + 当前指令发给 LLM

### 2.3 Token 计数: 启发式估算

Agent Xi 不用 tiktoken，用启发式:
- 中文: 1.5 token/字
- 英文: 1.3 token/词
- 安全系数: 1.1x（宁可多估不少估）
- 每条消息固定开销: 4 token（role 标记等）

**教训:** 精确计数需要引入 tokenizer 依赖（不同模型还不一样），启发式 + 安全系数在实践中够用。真正需要精确计数的场景（比如刚好卡在窗口边界）极少出现。

### 2.4 记忆注入位置

召回的记忆拼在 system prompt 尾部，不是作为单独的 user/assistant message。

**原因:**
- system prompt 中的内容注意力权重最高
- 作为 message 会被截断策略影响（可能被丢弃）
- 作为 system 的一部分，只要 system 不被截断（不会），记忆就在

**代价:** 记忆和 system prompt 竞争 token 预算。如果召回内容太多，会挤压历史消息空间。解决: 限制 recall top_k=3，每条记忆一行。

## 三、长程任务管理

### 3.1 ReAct 循环与工具调用

```
用户输入 → LLM 思考 → 调用工具 → 结果反馈 → LLM 再思考 → ... → 最终回复
```

**安全阀:** 最多 5 次迭代（硬编码）。防止 LLM 陷入"调工具→不满意→再调→还不满意"的死循环。

**工具结果处理:**
- 完整结果存入 history（不截断）
- 发给前端的预览截断到 200 字符
- 下一次 build_request 时，工具结果和其他消息一样参与截断竞争

**math_agent 适用点:** 数学建模的求解 Agent 可能需要多步计算（调 SymPy/NumPy）。同样需要迭代上限。建议: 求解 Agent 最多 8 次工具调用（数学推导步骤可能比通用对话多），超限后强制输出当前最佳结果 + 说明卡在哪里。

### 3.2 会话持久化与恢复

**策略:** 每轮对话结束后立即持久化（不是等断开连接）。

```
每轮结束 → save_session() → JSON 文件（原子写入: tmp + rename）
断开连接 → 触发记忆提取（service.md 重写）
重新连接 → 带 session_id → 从 JSON 恢复历史 → 继续
```

**关键细节:**
- 原子写入防损坏: 先写 .tmp 再 rename，进程崩溃不会留下半截文件
- 持久化上限 200 条消息（防磁盘膨胀）
- session_id 校验: `^[A-Za-z0-9_-]{8,64}$`（防路径穿越攻击）
- 损坏文件优雅降级: 返回空历史 + 日志警告，不崩溃

**math_agent 适用点:** 数学建模是典型的长程任务（可能跨越多天）。需要:
- 问题状态文档持久化（不只是聊天历史）
- 阶段检查点: 完成"问题分析"后保存，完成"模型选择"后保存...
- 断点续做: 用户隔天回来，从上次阶段继续，不是从头来

### 3.3 工具确认机制

敏感工具（写文件、执行命令）需要用户确认:

```
LLM 要调工具 → 检查安全级别 → SENSITIVE/DANGEROUS → 发确认请求给前端
→ 用户点允许/拒绝 → 60秒超时自动拒绝 → 继续/跳过
```

**设计决策:** 超时 = 拒绝（安全第一）。不是超时 = 允许。

### 3.4 多智能体上下文隔离

Agent Xi 是单 Agent，但 Session 隔离模式可推广:
- 共享: LLM 客户端、记忆管理器、工具注册表
- 隔离: 每个 Session 有独立的 Brain（独立历史）

**math_agent 推广:**
- 共享: 知识库、Embedding 客户端、LLM 客户端
- 隔离: 每个子 Agent 有独立的 system prompt + 工作上下文
- Orchestrator 持有全局状态（问题文档、阶段进度），子 Agent 只看到自己需要的切片

## 四、踩坑记录

### 4.1 记忆系统

| 坑 | 原因 | 解决 |
|---|---|---|
| 一次对话 14 条重复记忆 | 每轮都提取 + 无去重 + HMR 重载触发多次 | 改为会话结束时整体重写 |
| module-level os.environ 读空值 | import 在 load_dotenv() 之前执行 | 用 lazy getter 函数延迟到请求时读取 |
| LLM 重写时"发挥过度" | prompt 没有字数限制和逃逸条件 | 加 300 字预算 + "无新认知则原样返回" |
| 重写后丢失关键信息 | 没有快照，无法回滚 | 每次重写前存 .bak，保留 20 份 |

### 4.2 上下文管理

| 坑 | 原因 | 解决 |
|---|---|---|
| 工具结果太长撑爆上下文 | 未对工具输出做截断 | 下一轮 build_request 时自然截断（贪心策略） |
| system prompt 太长挤压历史 | 人设文件 + 工具列表 + 记忆全塞 system | 保底 1000 token 给历史；工具列表只放 name+description |
| 恢复会话后上下文断裂 | 持久化 200 条但 context 只发 100 条 | 可接受——最近 100 条足够；更早的靠情景记忆召回 |

### 4.3 部署与运行时

| 坑 | 原因 | 解决 |
|---|---|---|
| Docker 内 .env 不生效 | load_dotenv 找 CWD 下的 .env，Docker WORKDIR 要对 | 确保 WORKDIR=/app 且 .env 挂载到 /app/.env |
| 国内服务器拉 Docker Hub 超时 | 网络问题 | 配置 registry-mirrors + 去掉非必要 apt-get |
| WebSocket 不走 HTTP 中间件 | Starlette BaseHTTPMiddleware 只拦截 HTTP | WS 端点内手动校验 cookie |

## 五、math_agent 设计建议

### 5.1 记忆层设计

```
┌─────────────────────────────────────────────────┐
│  工作记忆 (Working Memory)                        │
│  = 当前问题状态文档，每轮更新                      │
│  类比 Agent Xi 的 service.md                     │
│  内容: 问题理解、已选模型、当前进度、待解决子问题    │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│  情景记忆 (Episodic Memory)                      │
│  = 历史建模经验向量库                             │
│  "上次类似题用了灰色预测，效果不好"                │
│  "用户偏好先看图再建模"                          │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│  知识库 (Knowledge Base)                         │
│  = 方法卡片 + 真题论文 + 模板（预建，只读）        │
│  静态 embedding + 元数据过滤                     │
└─────────────────────────────────────────────────┘
```

### 5.2 上下文策略

- Orchestrator: 问题文档(全文) + 当前阶段指令 + 相关子Agent最新输出
- 子 Agent: 角色定义 + 问题文档(相关切片) + 知识库检索结果 + 具体指令
- 不要把全部历史发给每个子 Agent——它们只需要"当前任务相关的上下文"

### 5.3 长程任务检查点

```
问题分析完成 → checkpoint_1.json (问题理解 + 约束 + 目标)
模型选择完成 → checkpoint_2.json (+ 模型方案 + 理由)
求解完成     → checkpoint_3.json (+ 代码 + 结果)
验证完成     → checkpoint_4.json (+ 灵敏度分析 + 改进方向)
论文完成     → checkpoint_5.json (+ 全文)
```

每个 checkpoint 是独立可恢复的。用户关掉电脑明天回来，从最新 checkpoint 继续。

### 5.4 多 Agent 通信

Agent Xi 的教训: 记忆注入 system prompt 比作为 message 更稳定。推广到多 Agent:
- 子 Agent 的输出返回给 Orchestrator 时，做结构化摘要（JSON），不是原始全文
- Orchestrator 把摘要写入问题文档（工作记忆），而不是追加到聊天历史
- 下一个子 Agent 从问题文档读取上游结果，不是从聊天历史中翻找

## 六、可直接复用的代码模式

### 6.1 整体重写 prompt 模板

```python
REWRITE_PROMPT = """你是一个记忆管理器。根据本次对话，重写下面的状态档案。

## 当前档案
{current_doc}

## 本次对话
{conversation}

## 规则
1. 过滤标准: "下次醒来不知道这件事，会做错事吗？" 答 yes 才写
2. 这是状态档案，不是日志。记录"是什么"，不是"发生了什么"
3. 合并重复，删除过时，300字以内
4. 如果没有新认知，原样返回当前档案
5. 保持 Markdown 格式，第一行是 # 标题

## 好的例子
- "用户是 TS/React 前端 + Python/AI 后端开发者"（稳定事实）
- "用户偏好先规划再动手，反感过度工程"（行为模式）

## 坏的例子
- "今天用户问了 Docker 部署"（日志，不是状态）
- "用户说了'可以'"（无信息量）

输出完整的新档案:"""
```

### 6.2 贪心截断

```python
def select_within_budget(messages, budget_tokens, count_fn):
    selected = []
    used = 0
    for msg in reversed(messages):
        cost = count_fn(msg)
        if used + cost > budget_tokens and selected:
            break
        selected.append(msg)
        used += cost
    selected.reverse()
    return selected
```

### 6.3 原子写入

```python
def atomic_write(path: Path, content: str):
    tmp = path.with_suffix(".tmp")
    tmp.write_text(content, encoding="utf-8")
    tmp.replace(path)  # 原子操作，崩溃不留残骸
```
